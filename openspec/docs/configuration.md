# Configuration — OpenSpec

OpenSpec configuration is split across three levels: global CLI config,
per-project `openspec init` options, and per-change `.openspec.yaml` files.

---

## Global CLI config

```bash
# View current global config
openspec config

# Set a value
openspec config set <key> <value>

# Set the workflow profile (selects which extra slash commands are available)
openspec config profile
```

### Profile

The profile controls which workflow commands are activated beyond the core
`/opsx:propose`, `/opsx:apply`, `/opsx:explore`, `/opsx:archive`.

```bash
openspec config profile
```

Prompts you to choose. After selecting, apply to the current project:

```bash
openspec update
```

---

## Per-project init options

`openspec init` accepts flags to control what gets generated:

```bash
openspec init --tools claude     # generate CLAUDE.md instructions
openspec init --tools opencode   # generate AGENTS.md instructions
openspec init --tools codex      # generate AGENTS.md instructions
openspec init --tools cursor     # generate .cursor/rules instructions
openspec init                    # auto-detect installed tools
```

The generated instruction files tell the AI agent how to use OpenSpec. Run
`openspec update` inside a project to regenerate them after upgrading.

## Schemas

A schema defines the artifact workflow for a change — which artifacts to
create, in what order, and what `apply` requires before implementation starts.

List available schemas:

```bash
openspec schemas
```

Example output:

```
SCHEMAS
  spec-driven    Default. Creates proposal, specs, design, tasks.
                 Apply requires: tasks
```

The default schema is `spec-driven`. Community schemas are available as
standalone npm packages — see the
[OpenSpec customization docs](https://github.com/Fission-AI/OpenSpec/blob/main/docs/customization.md)
for the catalog.

### Using a community schema

```bash
# Install the schema package
npm install -g @some-org/openspec-schema-tdd

# Register it with OpenSpec
openspec schema install @some-org/openspec-schema-tdd

# Use it when proposing
openspec new change my-feature --schema tdd
```

## Per-change `.openspec.yaml`

Each change directory contains an `.openspec.yaml` managed by the CLI:

```yaml
name: add-dark-mode
schema: spec-driven
created: "2026-06-03T10:00:00Z"
status: in-progress
```

Do not edit this file manually. Use the CLI to create (`openspec new change`),
inspect (`openspec status`), and archive (`openspec archive`) changes.

See [`config/openspec.yaml.example`](../config/openspec.yaml.example) for an
annotated example.

## Telemetry opt-out

OpenSpec collects anonymous command names and version numbers. No arguments,
file paths, content, or PII are collected. Automatically disabled in CI.

```bash
# Permanent opt-out (add to ~/.bashrc or ~/.zshrc)
export OPENSPEC_TELEMETRY=0

# Standard DO_NOT_TRACK also works
export DO_NOT_TRACK=1
```

---

## Environment variables

| Variable | Default | Description |
|----------|---------|-------------|
| `OPENSPEC_TELEMETRY` | `1` | Set to `0` to disable telemetry |
| `DO_NOT_TRACK` | unset | Set to `1` to disable telemetry (standard) |
| `NO_COLOR` | unset | Set to any value to disable colour output |
